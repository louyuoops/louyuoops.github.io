字符编码

ctf刷题过程突然对字符编码转换有点疑惑，查询资料总结一下常见的字符编码

什么是字符编码

字符编码词条的解释为[百度词条]：

字符编码（英语：Character encoding）也称字集码，是把字符集中的字符编码为指定集合中某一对象（例如：比特模式、自然数序列、8位组或者电脉冲），以便文本在计算机中存储和通过通信网络的传递。常见的例子包括将拉丁字母表编码成摩斯电码和ASCII。其中，ASCII将字母、数字和其它符号编号，并用7比特的二进制来表示这个整数。通常会额外使用一个扩充的比特，以便于以1个字节的方式存储。

作为计算机相关专业的学生，也许最开始入学学习编程时，都会接触到ASCII编码这个玩意，说得通俗点，这个就是将英语体系中的常见字符用数字编号，然后方便计算机去理解这是一个什么字符。

但是ASCII编码作为早期计算机的标准字符编码，在后续的发展过程中不再适用于所有国家的字符集，所以后续逐渐发展出很多字符编码，GB2312，BIG5，unicode，utf-8，utf-16等。

#️⃣本篇文章主要学习ASCII编码到GB2312，unicode到uutf-8编码的发展以及特点

ASCII码

全称 American Standard Code For Information Interchange.

依稀记得大学时买的《C Primer Plus》翻开书第一页便印着ASCII码表，当时第一印象便是以后书里面便会用前面标注的数字来代替相应的字符。

由于计算机是一个二进制的电路集成，所以计算机对字符理解只有0和1的组合，一个0或1称为一个bit，ASCii码是使用一个字节(8 bit)来表示一个字符。而一个字节表示表示的数是有上限（有符号整数）的，容量范围是0-128。其中ASCII 表中的前 32 个字符是不可打印的控制代码，用于控制打印机等外围设备。

但是127个字符显然无法保存所有国家的字符，后来在ASCII码的基础上补充了ASCII扩展码，也就是128-255表示的一些奇奇怪怪的字符。

GB2312，GBK编码

当计算机流入了具有上下五千年文化的大中国之后，突然有个问题，咱也不太熟悉英文，该怎么让计算机理解我们那么多的文字呢。显然一个字节扩展也只能表示255个字符，当时有个ANSI编码，意思就是用两个字节来延伸扩展字符编码；它并不是一个具体的字符编码集，而是一个通俗的标准吧（比如我国在ANSI基础上的编码为GB2312，日文在ANSI编码基础上的编码为JIS），又称为MBCS(Multiple-Bytes Character Set)多字节字符集。所以那时在ANSI编码的基础上发展处了适用于汉语的GB2312编码；而GBK编码是由于后来发现GB2312编码集并不能表示更多的汉字了，再次基础上有了一个扩展编码就是GBK。

GB2312-1980编码

GB2312-1980是1980发布的一般标准，GB2312使用一个94x94的矩阵来存储字符信息，其中每一行称为一个“区”，每一列称为一个“位”，矩阵中的每一个元素就是用区位码来表示。如第一个汉字“啊”出现在第16区的第1位上，其区位码为1601。

分区范围 符号类型
第01区 中文标点、数学符号以及一些特殊字符
第02区 各种各样的数字序号
第03区 半角西文字符
第04区 日文平假名
第05区 日文片假名
第06区 希腊字母表
第07区 俄文字母表
第08区 中文拼音字母表
第09区 制表符号
第10－15区 无字符
第16－55区 一级汉字（以拼音字母排序， 共3755个）
第56－87区 二级汉字（以部首笔画排序， 共3008个）
第88－94区 无字符

https://www.bejson.com/document/gb2312/

GB2312虽然用区位码来表示字符在字符矩阵中的位置，但是并不是存储为区位码，比如词条中的例子，

例如汉字‘珀’在GB2312中的区位码为7174，其两字节表示形式为71，74；而两个西文字符‘GJ’的存储码也是71,74。这种冲突将导致在解释编码时到底表示的是一个汉字还是两个西文字符将无法判断。

所以GB2312存储时是将区码和位码分别加上A0H得到存储码，以71，74为例

区位码 区位码（oct） 区码转换 位码转换 存储码
71，74 474AH 47H+A0H=E7H 4AH+A0H=EAH E7EAH

存储码又称为汉字内码，其实是由区位码进化到国标码(交换码)，再由国标码进化得到内码(存储码)

  • 国标码=区位码+32

  • 内码=国标码+128

内码中每个字节最高位置都为1（就是加128）。

GBK编码

GB2312编码只收集了6000多个汉字，再加上少数民族文字，显然是不够用，后来在GB2312基础上扩展有了GBK编码，GBK主要利用了GB2312中没有使用到的码位，并且兼容GB2312。

GBK编码最大的特点是将2字节的汉字字符和1字节的英文字符并存于一套字符集中。也就是说原本的ASCII编码集中127以下的字符是按照一字节存储的，其他则为双字节编码。

其中红色部分是ASCII码和原本GB2312中的编码，绿色为GBK扩展部分，紫色为用户自定义编码范围

http://106.14.124.170/Encoding/GBK

unicode编码

unicode编码是在ISO组织带领下，将所有字符用双字节表示发展出来的一种字符编码（UCS Universal Multiple-Octet Coded Character Set），对于ASCII编码来说，高位字节不足的补0；其实这就出现了一个问题，假如一个文档是全英文的，那么存储空间几乎就是多了一倍，这就浪费了一倍的空间；不过好处就是所有字符编码统一了。

但是严格意义来说，unicode编码只是将字符编码了，也就是规定了字符的二进制编码，并没有规定应该如何存储字符；GB2312用区位码规定了每个字符的编码，但是也具体给出了使用存储码的方式来存储字符，用来区分和其他编码存储的不同。

后来在万维网的发展下，就急需一个普适的传输编码，就有了UTF编码（UCS Transfer Format）。

https://unicode-table.com/cn/

utf编码

utf编码是一个统称，主要包括utf-8，utf16，utf32编码。其中使用最为广泛的是utf-8。

UTF-8

utf-8编码是一种变长编码，使用1-4个字节存储字符。

UTF-8中分为单字节存储和多字节存储；

  • 单字节存储，即保存ASCII字符集中127及以下的字符，所以utf-8兼容ASCII，解决了unicode中双字节存储浪费空间的问题。

  • 多字节存储，比如存储一个3(n)字节长的字符，第一个字节的前3(n)位为1，第2第3个字节(后续的字节)的前两位都为10，其余部分分别填入该字符的unicode编码一部分；加入顺序是从低位到高位，（逐次从unicode从的低位取6位填到utf-8的低位字节后6位中，直到取完，不足补0，如下面例子中最后0100补不齐5位，最后补了0）

UTF-16

utf-16编码规则：

  1. 对于 Unicode 码小于 0x10000 的字符， 使用 2 个字节存储，并且是直接存储 Unicode 码，不用进行编码转换

  2. 对于 Unicode 码在 0x10000 和 0x10FFFF 之间的字符，使用 4 个字节存储，这 4 个字节分成前后两部分，每个部分各两个字节，其中，前面两个字节的前 6 位二进制固定为 110110，后面两个字节的前 6 位二进制固定为 110111, 前后部分各剩余 10 位二进制表示符号的 Unicode 码 减去 0x10000 的结果

  3. 大于 0x10FFFF 的 Unicode 码无法用 UTF-16 编码

UTF-32

UTF-32 是固定长度的编码，始终占用 4 个字节，足以容纳所有的 Unicode 字符，所以直接存储 Unicode 码即可，不需要任何编码转换。虽然浪费了空间，但提高了效率。

关于字节序

utf-16因为使用双字节打包存储，需要分别大端小端序，大端小端这好理解，测试一下，分别以小端存储“Untitled-1”和大端存储“Untitled-2”

1>>> fp = open("Untitled-1", "rb")
2>>> print(fp.read())
3b'\xff\xfeu\x00t\x00f\x00-\x001\x006\x00Km\xd5\x8b'    // fffe开头是小端
4>>> fp = open("Untitled-2", "rb")
5>>> print(fp.read())
6b'\xfe\xff\x00u\x00t\x00f\x00-\x001\x006mK\x8b\xd5\x002'  // feff开头是大端
7>>>

utf-32也存在大端小端序。

*但是UTF-32和UTF-16有一个很严重的问题——和C语言不兼容。因为C语言中_*0000 0000**_表示字符串结尾，而UTF-32和UTF-16中有很多字符高位都是0，和字符串结尾冲突了。

上述所属的不兼容主要是由于utf-16和utf-32直接存储unicode编码导致，unicode编码中存在ASCII字符集中的127及以下的字符高位字节补0操作。

问题

1: 为什么utf8不分大小端序？

答：由于UTF-8的设计，一个字符对应的各个字节，是能够区分出 哪个是数据高位、哪个数据低位的，所以不需要。
